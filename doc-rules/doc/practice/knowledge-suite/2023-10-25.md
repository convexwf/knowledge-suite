# knowledge-suite Journal - 2023-10-25

## 14:40:08 style: rebalance side panel typography

After the previous workflow simplification commit, the user still felt the
side panel typography was uncomfortable and specifically called out a mismatch
between perceived text size and emphasis: the UI text looked pale, the saved
list felt more prominent than the reading surface, and the actual rendered
preview still somehow read as too small despite earlier size increases. The
user also asked me to compare against the period before commit
`c990b4a02efc65c67f5a6eb68f098fc22b6ec3ac` instead of continuing to tweak in
the dark.

I inspected `c990...^:apps/knowledge-web-clipper/public/side-panel.css` and
confirmed the older version did not rely on larger numbers so much as stronger
contrast and clearer hierarchy. Before the later redesign layers were added,
preview body copy was `14px`, saved titles were `13px`, saved metadata was
`11px`, and colors used direct values like `#1f2933` and `#526174`. The
current side panel had drifted into a visually softer state because warm panel
backgrounds, tokenized muted text colors, enlarged saved titles, and generous
line heights had all been stacked together. That combination made the preview
feel less anchored even though the raw font size was technically larger.

Based on that comparison, I chose not to keep scaling everything upward.
Instead I rebalanced the hierarchy in
`apps/knowledge-web-clipper/public/side-panel.css` around three principles:
make the rendered preview the darkest element again, reduce saved and batch
text so it sits below the preview rather than competing with it, and tighten
supporting UI copy so the panel no longer feels washed out. Concretely, I
removed the extra `body` font-size override, darkened helper text from token
muted values to explicit `#526174`-family tones, reduced saved and batch title
sizes back down to `15px`, reduced metadata to `13px`, and brought parser copy
closer to the older proportions.

For the preview itself, I kept it larger than the historical baseline but much
closer to a stable reading scale: `17px` body text with `1.72` line height,
`26/22/19/17` heading scale, and `14px` code. This preserves the intent of the
recent readability push without letting the rest of the side panel inflate
past the content it is meant to support. I also darkened blockquote and general
preview text away from the softer token palette so the reading surface feels
more solid.

The main trade-off here was intentionally shrinking several pieces of UI that
had just been enlarged in the previous iterations. That might sound like
backtracking, but the older commit comparison made it clear the real issue was
not “everything needs to be bigger”; it was “the preview needs to be the most
legible thing.” Re-establishing hierarchy was more important than preserving
every earlier size increase.

Verification used `npm run build -w @uknowledge/knowledge-web-clipper`, which
completed successfully after the CSS changes. The fix stayed scoped to
`side-panel.css`, with this journal entry documenting the rationale from the
historical comparison.

```text
style: rebalance side panel typography

- restore stronger text contrast and reduce washed-out side panel copy
- make preview text hierarchy more prominent than saved and batch list entries
- tune the reading scale toward the pre-c990 baseline with a calmer size ladder
```

## 14:53:38 fix: restore preview body typography hierarchy

The user was still unconvinced by the last typography pass and asked for a
true end-to-end test using a real article URL instead of more CSS inspection.
The specific suspicion was that the rendered article body might still be
smaller than the side-panel list typography even though the stylesheet looked
like it should favor the preview surface.

I verified that suspicion with an actual browser rendering pass. First I
fetched the Freedium mirror of the supplied article and copied representative
body paragraphs into a local test fixture that reused the current
`apps/knowledge-web-clipper/public/side-panel.css`. I then rendered that
fixture in headless Chrome and dumped computed styles from the live DOM. The
result exposed the real bug: `.preview-output` itself was `17px`, but the
actual article paragraphs inside it were only `13px` because the global `p`
rule overrode them. At the same time, `.saved-title` was `15px`, so the user's
visual complaint was factually correct: saved-list titles were larger than the
main reading paragraphs.

The fix stayed in `apps/knowledge-web-clipper/public/side-panel.css`. I added
explicit preview-content inheritance rules for `.preview-output p`, `ul`, `ol`,
`li`, `blockquote`, and `blockquote p`, forcing them to inherit the preview
surface font size, line height, and color instead of falling back to the
global side-panel body copy rules. I intentionally left saved-list sizing alone
in this pass, because the end-to-end measurements already showed that once the
preview paragraphs truly inherit `17px`, the hierarchy flips back to the
desired direction without needing another broad restyle.

I also created and then removed a temporary local HTML fixture used only for
verification, so the final code change remains scoped to the real stylesheet.
That fixture was never meant to ship; it existed solely to measure computed
styles against realistic article content.

Verification was stronger than in previous rounds. I ran
`npm run build -w @uknowledge/knowledge-web-clipper`, then rendered the local
fixture in headless Chrome and checked the measured DOM values twice:
before the fix `.preview-output p` was `13px` while `.saved-title` was `15px`;
after the fix `.preview-output p` became `17px` while `.saved-title` remained
`15px`. That confirmed the root cause and the correction directly.

```text
fix: restore preview body typography hierarchy

- make preview paragraphs and list content inherit the intended reading scale
- fix the global paragraph rule overriding rendered article body text
- verify with headless Chrome using the supplied Freedium article content
```
