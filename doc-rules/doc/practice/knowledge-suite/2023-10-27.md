# knowledge-suite Journal - 2023-10-27

## 15:06:49 fix: streamline side panel feedback flow

The user reported three concrete follow-up issues in the clipper side panel.
First, clicking save still pushed the `Preview` and `Saved` surfaces downward
because the status detail remained part of the toolbar layout instead of acting
like transient feedback. Second, the earlier request about the diagnostics
chevron had been misinterpreted: the user wanted the down-arrow to remain
available, but to consume almost no space, not to disappear entirely. Third,
the real-webpage typography verification I had just done manually needed to be
carried into the existing automated test framework so future UI refactors do
not regress the same preview-vs-list hierarchy again.

Analysis broke into two parts: product behavior and test stability. On the
product side, the toolbar status detail was the direct cause of the preview
surface jumping during save and delete flows, so that detail needed to leave
normal document flow entirely. On the diagnostics side, the right compromise
was a zero-height separator row with a tiny chevron button that visually sits
on the divider instead of reserving a real bar in the layout. On the test
side, I first integrated assertions into the existing Playwright extension flow
for save toast visibility, preview-top stability, compact diagnostics layout,
and computed preview-vs-saved font sizes using the extension against a live
local article. While doing that, I found the old tail of the E2E script was
flaky for an unrelated reason: once the side panel is opened as an extension
tab in Playwright, delete-style actions can accidentally re-target the
extension page itself as the active tab. That is a historical limitation of
the test harness shape rather than this UI change, so I explicitly narrowed
the E2E scope to the side-panel behaviors this pass is responsible for.

The implementation touched four files. In
`apps/knowledge-web-clipper/public/side-panel.html`, I removed the inline
status-detail slot from the toolbar, reintroduced a dedicated but tiny
diagnostics toggle row, and added a toast container at the edge of the panel.
In `apps/knowledge-web-clipper/src/side-panel.ts`, I wired status updates so
the pill still reflects current state while transient detail messages appear as
auto-dismissing toast feedback, and I restored the diagnostics disclosure
toggle behavior with correct aria state. In
`apps/knowledge-web-clipper/public/side-panel.css`, I styled the toast as a
fixed overlay so save feedback no longer resizes the reading surface, and I
made the diagnostics chevron effectively zero-footprint by putting it on a
zero-height row with an absolutely positioned compact button. Finally, I added
`apps/knowledge-web-clipper/tests/side-panel-styles.test.ts` for stylesheet
guards and updated `scripts/e2e-extension.mjs` so the existing browser E2E
asserts the real preview typography hierarchy, the toast behavior, and the
compact disclosure geometry.

Two decisions are worth preserving. The first is that save-state detail now
lives in toast rather than in persistent chrome because the reading surface
matters more than keeping every message permanently visible. The second is that
the E2E script now triggers an explicit refresh after switching target tabs in
the real-page flow; relying on auto-refresh inside a tab-opened extension page
looked elegant but was materially less stable than asserting the rendering
result after a deterministic refresh.

Verification was intentionally stronger than a normal CSS pass. I ran
`npm test -w @uknowledge/knowledge-web-clipper`, which passed all 17 tests
including the new stylesheet assertions. I ran
`npm run build -w @uknowledge/knowledge-web-clipper`, which completed
successfully. I also ran `npm run e2e:extension` after updating the extension
flow, and it passed with the new live assertions for toast visibility,
preview-layout stability, compact diagnostics geometry, and computed
`#preview-output p = 17px` versus `#saved-list .saved-title = 15px`.

The reusable pattern here is to keep subjective UI complaints grounded in live
DOM measurements when possible. In this side panel, typography bugs were easy
to misread from CSS alone because component-level tokens and global element
rules interacted in non-obvious ways. Locking the important ratios and layout
behaviors into both unit-style checks and a real extension browser test gives a
much better guardrail for the next round of design iteration.
