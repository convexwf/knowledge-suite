## P1: Stale job recovery, collection navigation API, reader context, retry

Implemented P1 steps 3-7:

**Backend:**
- Added `recoverStaleJobs()` in store.ts — marks all `queued`/`running` batch jobs as `failed` on server startup. Called in `buildServer()` after `store.ensure()`.
- Added `getCollectionNavigation()` in store.ts — given a collectionId and docId, finds the previous and next items by `order_index`.
- Added `GET /api/collections/:collectionId/navigation` route (defined before `:collectionId` catch-all to avoid route conflict).
- Added `resetFailedBatchItems()` in store.ts — resets failed batch items in a job back to `pending` state, clearing error info.
- Added `POST /api/batch/jobs/:jobId/retry` endpoint — resets failed items, sets job back to `queued`, and re-runs the worker.

**Frontend API client:**
- Added `collectionNavigation()` and `retryBatchJob()` methods.

**Reader page (reader.html + reader.ts):**
- Added fourth fact card "Collection" showing the collection name(s) the document belongs to.
- Added collection navigation bar between header and layout with `Previous in Collection` / `Next in Collection` buttons.
- Added `loadCollectionContext()` — fetches item detail for collectionIds, then loads collection names and navigation data.
- Added `navigateInCollection()` — navigates to prev/next doc in the collection.
- Collection nav buttons are disabled when there's no prev/next item.

**Batch progress modal:**
- Added "Retry N Failed" button when there are failed items. Clicking it retries only the failed items and restarts polling.

**CSS:**
- `reader.css`: Updated `.reader-facts` grid to 4 columns. Added `.collection-nav` styles.

## 14:49:06 style: refine reader annotation surfaces

The user shifted focus from the annotation index page to the reader itself and
called out two quality issues: the annotation sidebar still looked too plain,
and clicking a highlight or note in the article produced a popup that felt more
like a utilitarian tooltip than a well-designed reading companion. The request
was to redesign both surfaces together and then sync the resulting decisions
back into the technical design so the document remains authoritative.

I treated this as a reader-surface refinement rather than an annotation-data
change. The existing sidebar only had a flat summary, simple filter buttons,
and a one-line list that fused type and body text together. That made every
annotation look like the same generic row. The popup had the same issue: it
showed a title, one text block, and a delete button, but it did not express the
underlying annotation object in a structured way. The redesign goal was to make
`highlight`, `note`, and `summary` feel like different kinds of content while
staying compact enough for a reading workspace.

In `src/reader.ts`, I rewrote `renderAnnotationSidebar()` so it now builds a
proper card flow with explicit `type / section / reference / body / meta`
structure. The sidebar summary also became more informative: it now shows total
annotations, a type breakdown, and a clearer empty state. Filter chips carry
counts directly, and the list now prioritizes annotations belonging to the
currently visible section by deriving the nearest visible `data-section-id`
from the reader DOM instead of relying on a separate missing state variable.
That keeps the most relevant notes near the top while still preserving the full
document list.

I also rewrote `showAnnotationPopup()` so inline annotation clicks open a
context card rather than a bare tooltip. The new popup has a small type header,
section and time metadata, an optional reference block, a main content block,
and a bottom action row. Highlights can still surface their color, summaries
now read like stronger conclusion cards, and orphaned items can show a light
badge instead of hiding that state.

In `public/reader.css`, I restyled both surfaces to match that structure. The
sidebar cards now have real spacing, type pills, reference blocks, and
secondary metadata instead of a single compressed line. The popup now uses a
larger rounded card, better internal hierarchy, and lighter destructive action
treatment. The intent is that the reader-side annotation UI should feel like a
companion note system, not like debug metadata glued next to the article.

Verification covered both the TypeScript pass and the final frontend bundle. I
ran `npm test -w @uknowledge/knowledge-web-clipper`, which passed all tests. An
initial build failed because I mistakenly referenced a non-existent
`activeHeading` state, so I replaced that with a DOM-based `currentVisibleSectionId()`
helper and reran `npm run build -w @uknowledge/knowledge-web-clipper`, which
then passed. That bug is worth preserving as a reminder to ground sidebar state
in actual reader DOM visibility when no dedicated heading-selection model
exists.

```text

- redesign the reader annotation sidebar into structured content cards
- upgrade inline annotation popups into richer contextual note cards
- prioritize annotations near the current visible section in the reader
```

## Fix: reader collection context not showing for docId-only paths

When the reader was opened from the side panel's saved clips list, it passed only `docId` (no `itemId`). The `loadCollectionContext()` function required `currentItem.itemId` and immediately returned `-` without looking up collections.

Fix:

- Added `getCollectionsByDocId()` in store.ts — queries `collection_items` joined with `collections` directly by `doc_id`.
- Added `GET /api/collections/by-doc` route (defined before `:collectionId` to avoid parameter capture).
- Added `collectionsByDoc()` API client method.
- Rewrote `loadCollectionContext()` to handle two paths: item-based lookup and doc-based lookup.
- Navigation buttons are now enabled for both paths.

```text

- add getCollectionsByDocId() store method
- add GET /api/collections/by-doc endpoint
- add collectionsByDoc() API client method
- update loadCollectionContext() to handle docId-only case
```
