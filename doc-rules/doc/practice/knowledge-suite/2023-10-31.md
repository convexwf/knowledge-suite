# knowledge-suite Journal - 2023-10-31

## 16:22:03 style: rebalance annotation workspace

The user felt the annotation page looked top-heavy and specifically pointed out
that the overview stat tiles could overflow their bounds. The broader request
was not just to patch the overflow, but to rethink the whole page structure and
bring the technical design back in sync with the improved direction.

I reviewed the current page as a layout problem first. The left navigation had
become too visually heavy because the header copy plus stat cards occupied a
large share of a narrow column, while the right side opened with a relatively
light title block. That imbalance made the page feel like the navigation was
the main feature and the annotation stream was secondary. The overflow problem
came from the right-side stat cards trying to render raw type labels directly
as large values, which is unstable once a type label is longer than a short
single word.

The implementation intentionally shifted visual weight back to the right side.
In `public/annotations.html`, I kept the two-column structure but simplified
the nav header markup so the top copy block could be styled more compactly. In
`src/annotations.ts`, I changed the document overview cards from “top three raw
types” to a more stable trio of `Total`, `Leading Type`, and `Orphaned`, which
prevents overflow while still surfacing the most useful document-level summary.
I also promoted the right-side header into a more obvious page title zone with
an `Annotation Index` kicker so the detail panel reads like the page’s main
workspace.

The biggest change was a full rewrite of `public/annotations.css`. The old file
still carried stacked generations of rules and made it hard to reason about the
final layout. I rewrote it into a cleaner structure with a compact left nav,
card-like document entries, lighter global stats, stronger right-side overview
cards, and a more editorial annotation stream. The document nav now reads like
a selector, not a dashboard. The detail side now has clearer hierarchy:
headline, overview cards, filter/action toolbar, then the annotation cards. I
also added clamped and wrapping-safe stat values so long content no longer
pushes outside card boundaries.

Two design decisions are worth preserving. First, the annotation page should be
read as a document-centric workspace, not a stats-first dashboard. Second,
overview cards should prefer stable semantic summaries over raw taxonomy labels
when there is a risk of text overflow or layout instability.

Verification included both the normal frontend checks and the implementation
scope review. I ran `npm test -w @uknowledge/knowledge-web-clipper` and
`npm run build -w @uknowledge/knowledge-web-clipper`, and both passed after the
rewrite. The changes in this commit are limited to `annotations.html`,
`annotations.css`, and `src/annotations.ts`.
