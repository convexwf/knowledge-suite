# knowledge-suite Journal - 2023-10-23

## 12:02:37 style: add shared clipper visual foundation

The user asked to start implementing the UI refresh in small phases and to stop
after each phase for a dated commit. I treated this first pass as the safest
foundation phase: do not reshuffle page structure yet, but make the extension
pages feel like the same product by introducing shared design tokens and a
common visual baseline that later phases can build on without fighting four
separate style systems.

I reviewed the current `knowledge-web-clipper` front-end layout before editing
and noticed the same pattern across `side-panel.css`, `items.css`,
`reader.css`, and `annotations.css`: each page duplicated base body, button,
panel, and border styling, and each used slightly different neutrals, radii,
and accent colors. The right move for Phase 1 was not a deep refactor of HTML
or TypeScript, but a shared static CSS layer plus local override blocks at the
end of each page stylesheet. That keeps the risk low while still establishing a
unified palette and typographic direction.

I added `apps/knowledge-web-clipper/public/styles/tokens.css`,
`base.css`, and `components.css`. `tokens.css` defines the new warm editorial
background, primary green accent, warm annotation tone, softer panel surfaces,
and shared radius/shadow scales. `base.css` centralizes the button, input,
heading, paragraph, focus, and body defaults. `components.css` provides the
shared treatment for primary buttons, danger buttons, menu surfaces, empty
states, and the status pill. I then updated
`apps/knowledge-web-clipper/scripts/copy-static.mjs` so the new `public/styles`
directory ships into `dist/styles` during extension builds.

For the page-level work, I did not delete the existing CSS blocks. Instead I
imported the shared styles at the top of `items.css`, `side-panel.css`,
`reader.css`, and `annotations.css`, then appended targeted override sections at
the bottom of each file. This phase therefore changes visual language without
changing markup contracts. `items.css` now uses the shared panel surfaces and
calmer batch bar. `side-panel.css` now pulls the status/header/tabs toward the
same palette and softens the preview surfaces. `reader.css` gets the strongest
editorial shift, including the reading serif stack, softer outline panel, and
warmer content cards. `annotations.css` now visually matches the same product
family rather than looking like an isolated admin page.

The main trade-off in this phase was duplication versus safety. A cleaner end
state would fully remove the older per-page base selectors, but that would make
the first commit much harder to review and risk subtle regressions in extension
pages that depend on current DOM structure. I intentionally chose additive
imports plus bottom-of-file overrides so later phases can progressively delete
old declarations once the new page-specific structures are in place.

For verification, I ran `npm run build -w @uknowledge/knowledge-web-clipper`
from the `knowledge-suite` repo. The build completed successfully, which
confirmed TypeScript output, static asset copying, and the new shared styles
directory all work together. I also checked the resulting git diff to confirm
only the intended shared-style foundation files and page CSS updates were
included. One important operational note: the dated-commit plan for this repo
returned `effective_date=2023-10-23` with `time_anomaly=true`, so this journal
entry is intentionally written under that historical date rather than today's
calendar date.
